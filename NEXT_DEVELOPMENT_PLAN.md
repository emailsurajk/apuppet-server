# aPuppet Kiosk Unattended Control - Next Development Plan

Date: 2026-04-17
Target: Fully owned rooted Android kiosk fleet with attended setup and unattended remote control operations.

## Confirmed decisions (2026-04-17)

- Primary deployment path is rooted/custom ROM/system-app.
- Unattended scope is all managed devices, enforced by server policy.
- Janus remains the media/signaling core; add a policy/enrollment control plane beside it.
- Operator identity, reason code, and full audit logging are mandatory for unattended sessions.
- Rollout approach is pilot first (10-20 devices), then phased expansion.
- Session ID and password generated by the Android app are pushed to an external credential server (not apuppet-server). The web-admin and operators retrieve them from that external server to connect to apuppet-server. apuppet-server itself has no session credential storage responsibility.

## 1) Scope and non-negotiables

This plan assumes:
- Devices are enterprise/company owned.
- Deployment is consented and policy governed.
- ROM/root/system control is available.

Out of scope:
- Hidden or unauthorized remote access.
- Consumer BYOD installation flow.

## 2) Current gaps

From current behavior:
- Accessibility permission prompt appears repeatedly.
- Session approval still requires local user action.
- Session ID/password is not auto-synced; credentials are still manually read from device screen by the operator.
- Future flow: app pushes session ID + password to an external credential server so operators can retrieve them without physical access to the device.
- Web admin is being hardened separately, but device-side unattended policy is not yet complete.

## 3) Target state

Each kiosk should:
- Auto-boot into kiosk agent.
- Auto-register with backend using device identity.
- Auto-establish remote-control readiness without local user interaction.
- Allow remote operators only through backend policy and audit controls.
- Support remote revoke and emergency disable.

## 4) Architecture decisions

## 4.1 Device mode
- Provision as dedicated managed device (Device Owner where possible).
- Keep app persistent with boot start + watchdog.
- Use lock task mode for kiosk UX.

## 4.2 Input control path
- Do not depend on Accessibility as the primary unattended control channel.
- Use privileged/root-capable input injection path suitable for your ROM/device family.
- Keep Accessibility optional as fallback for non-root debug builds only.

## 4.3 Screen capture path
- Standard MediaProjection flow requires user confirmation.
- For unattended kiosk, implement privileged capture path for rooted/system build.
- Keep feature flags per device family for fallback compatibility.

## 4.4 Session and identity
- Replace temporary session-only trust with persistent device identity.
- On first boot/provision:
  - Generate device keypair on device.
  - Register device to backend with signed enrollment.
  - Receive policy profile and control capability tokens.
- Rotate tokens automatically on schedule.

## 4.5 Server control plane
- Add policy engine fields per device:
  - unattended_enabled
  - allowed_operator_groups
  - allowed_time_windows
  - max_session_duration
  - require_reason_code
- Enforce policy at session creation and at periodic revalidation.

## 4.6 Session credential routing (external credential server)
- apuppet-server (Janus) is NOT responsible for storing or distributing session IDs or passwords.
- After the Android app creates a Janus session and generates a session ID + password, it pushes those credentials to a separate external credential server under your control.
- The web-admin (or any authorised operator client) polls or subscribes to the external credential server to retrieve the active session ID + password for a device.
- The operator then connects directly to apuppet-server using those credentials.
- apuppet-server only validates the Janus secret (api_secret) and the room PIN as it already does today; nothing else changes in Janus.
- Integration with the external credential server is a separate workstream (not in scope for apuppet-server development). apuppet-server changes needed:
  - None to Janus itself.
  - Android app: add a credential-push call after successful Janus room creation (push sessionId + password + device identity to external server endpoint).
  - Web admin: replace manual session ID/password entry form with a fetch from external server by device identity (to be implemented when external server is ready; keep manual entry as fallback).

## 5) Development phases

## Phase 1 - Baseline hardening (1 week)
Goals:
- Remove app-level manual accept gate in unattended mode.
- Add configuration flag for unattended mode in app + backend.
- Add structured audit events for connect/disconnect/command sessions.

Tasks:
1. Android app
	- Add unattended mode config source (remote policy + local fail-safe).
	- Split attended vs unattended flow in connection controller.
	- Persist device identity and registration state.
2. Server
	- Add device policy table/fields.
	- Add API endpoint for policy fetch and revocation status.
3. Web admin
	- Display device mode (attended/unattended).
	- Require operator reason before connecting in unattended mode.

Acceptance criteria:
- Unattended-enabled device can connect without local approval prompt from app logic.
- Full audit record exists for each remote session.

## Phase 2 - Privileged capability integration (2-3 weeks)
Goals:
- Eliminate recurring Accessibility consent dependency.
- Implement stable root/system path for input + capture.

Tasks:
1. Android platform integration
	- Implement privileged input injection module.
	- Implement privileged capture module.
	- Guard modules with startup self-checks.
2. App runtime capability manager
	- Detect available capability tier at runtime:
	  - Tier A: full privileged unattended
	  - Tier B: partial unattended (input only or capture only)
	  - Tier C: attended fallback
3. Backend reporting
	- Device posts capability tier and health status.

Acceptance criteria:
- Rooted target devices run unattended sessions with no local UI confirmations for app-managed actions.
- Capability tier visible in admin panel.

## Phase 3 - Policy, security, and resilience (1-2 weeks)
Goals:
- Make unattended safe, revocable, and operationally supportable.

Tasks:
1. Security
	- Device-backend mTLS.
	- Token expiration and refresh.
	- Certificate pinning in app.
2. Safety controls
	- Remote kill switch per device.
	- Session timeout + idle timeout.
	- Concurrent session prevention.
3. Reliability
	- Auto-reconnect on boot/network change.
	- Crash watchdog and service restart policy.

Acceptance criteria:
- Security review checklist passes.
- Kill switch disables remote control within defined SLA (for example < 30 seconds).

## Phase 4 - Fleet rollout (1 week pilot + staged expansion)
Goals:
- Validate in production-like conditions.

Tasks:
1. Pilot 10-20 devices with mixed network conditions.
2. Collect KPIs:
	- Session success rate
	- Median connect time
	- Crash-free hours
	- Reconnect success rate after reboot
3. Staged rollout with rollback gates.

Acceptance criteria:
- KPI thresholds met for 7 consecutive days before broad rollout.

## 6) Suggested API additions

### 6.1 apuppet-server (Janus side) - no new session storage APIs needed
- Janus REST API remains unchanged.
- apuppet-server does NOT expose session credential storage or retrieval endpoints.
- The only apuppet-server-side addition is ensuring the Janus api_secret and room PIN validation continue to work correctly when credentials arrive via the external server flow.

### 6.2 Android app - new outbound call (to external credential server)
- After Janus session and room creation: POST {externalServer}/api/device-sessions
  - Body: { device_id, janus_session_id, password, janus_server_url, created_at }
  - Authenticated with device identity token.
  - apuppet-server is not involved in this call.

### 6.3 External credential server APIs (separate workstream - not apuppet-server)
- POST /api/device-sessions          — device registers active session credentials
- GET  /api/device-sessions/{device_id} — operator client fetches active session for a device
- DELETE /api/device-sessions/{device_id} — device clears session on disconnect

### 6.4 apuppet-server control plane APIs (policy and enrollment - for later phases)
Device APIs:
- POST /api/devices/enroll
- POST /api/devices/heartbeat
- GET /api/devices/{id}/policy
- POST /api/devices/{id}/events

Session lifecycle APIs (if session orchestration moves server-side in a later phase):
- POST /api/sessions/request
- POST /api/sessions/{id}/start
- POST /api/sessions/{id}/end
- POST /api/sessions/{id}/revalidate

Security notes:
- Use signed device identity and short-lived operator tokens.
- Keep least privilege between operator roles.
- apuppet-server never holds or forwards raw session passwords; the external credential server handles that flow.

## 7) Data model sketch

### 7.1 apuppet-server / policy control plane (lives in apuppet-server)

Device:
- id
- serial_or_hardware_id
- public_key
- capability_tier
- unattended_enabled
- policy_version
- last_seen_at
- revoked_at

RemoteSession (lifecycle record only - no credential storage here):
- id
- device_id
- operator_id
- reason_code
- mode (attended|unattended)
- started_at
- ended_at
- status

AuditEvent:
- id
- session_id
- device_id
- operator_id
- event_type
- event_time
- metadata_json

### 7.2 External credential server (separate service - not apuppet-server)

DeviceSession (short-lived, cleared on disconnect):
- device_id
- janus_session_id    (the 8-char session ID used in Janus room)
- password           (the 4-char room PIN)
- janus_server_url   (which apuppet-server instance to connect to)
- created_at
- expires_at         (TTL-based auto-expiry)

## 8) Risk register

1. OEM/ROM fragmentation
- Mitigation: maintain compatibility matrix and feature flags.

2. OTA breakage on rooted hooks
- Mitigation: canary ring rollout and preflight tests.

3. Security misuse risk
- Mitigation: strict RBAC, audit alerts, kill switch, mandatory reason codes.

4. Operational drift
- Mitigation: continuous heartbeat and policy checksum validation.

5. External credential server availability
- If the external server is unreachable, the operator cannot retrieve the session credentials and the remote session cannot start.
- Mitigation: keep manual session ID/password entry as a fallback in web-admin; implement credential server with HA/replica; app retries push with exponential backoff and caches locally for re-push.

6. Credential leakage via external server
- Session ID + password in transit and at rest on external server must be protected.
- Mitigation: TLS-only transport; short TTL on stored credentials; restrict GET endpoint to authenticated operators; clear credentials immediately on session disconnect.

## 10) Test matrix

Functional:
- Boot -> auto-connect.
- Network drop/recover -> reconnect.
- Policy disable -> session blocked.
- Kill switch -> active session terminates.

Security:
- Expired token rejected.
- Revoked device rejected.
- Unauthorized operator rejected.

Resilience:
- Process crash recovery.
- Device reboot recovery.
- High latency network behavior.
